# 🎓 Noto — LectureNote AI

> **프로젝트명**: Noto (LectureNote AI)
> **핵심**: 학생이 수업 전에 교과서 PDF와 페이지 범위를 올리고, 수업 중 녹음을 켜두면, 수업 후 AI가 원본 PDF 위에 직접 필기를 해주는 웹 서비스

---

## 🎯 핵심 가치 제안 (Core Value Proposition)

기존 AI 노트 서비스들은 강의를 듣고 **"새로운 노트 문서"를 만들어준다**.

**Noto는 다르다.**
- 학생이 이미 갖고 있는 **교과서 PDF 위에 직접 주석을 달아준다**
- 학생은 자기 교과서를 열었을 때, AI가 선생님 강의 내용을 바탕으로 필기를 대신 해놓은 것처럼 보인다

---

## 📋 전체 사용자 플로우 (User Flow)

### 수업 전 (30초 세팅)
1. 웹 브라우저에서 PDF 업로드
2. "오늘 범위: 12페이지 ~ 18페이지" 입력
3. 녹음 시작 버튼 클릭

### 수업 중
- 브라우저 탭을 열어두거나 닫아도 됨 (백그라운드 녹음)
- 학생은 수업에만 집중

### 수업 후
1. 녹음 종료
2. AI 분석 시작 (수 분 소요)
3. 결과 화면: 원본 PDF 뷰어에서 결과 확인

---

## 🖼️ 결과 화면에서 보이는 것

### PDF 뷰어 (메인 영역)
- **원본 교과서 PDF가 그대로 표시됨**
- **[노란 하이라이트]**: 선생님이 강조한 용어나 문장
  - "이거 시험에 나와요", 반복 언급 등
- **[초록 하이라이트]**: 선생님이 추가 설명을 한 용어
  - 마우스를 올리면 툴팁으로 설명 표시
- **[회색 처리 / 표시 없음]**: 선생님이 특별히 언급하지 않은 부분

### 사이드바 (오른쪽)
- **[불확실 항목 목록]**: AI가 어디에 매핑해야 할지 확신하지 못한 내용들
  - 선생님이 PDF 범위 밖의 내용을 언급한 경우
  - AI가 용어를 정확히 인식하지 못한 경우
  - 학생이 직접 보고 수동으로 필기하거나 무시할 수 있음

---

## 🤖 AI가 수행하는 분류 작업 (3가지 버킷)

### 버킷 1 — 강조 (노란 하이라이트)
- 선생님이 "이거 중요해요", "시험에 나와요" 등 명시적으로 강조
- 같은 용어를 2회 이상 반복
- 목소리를 높이거나 속도를 늦추는 패턴 (STT 타임스탬프 기반 추정)

### 버킷 2 — 추가 설명 (초록색 + 툴팁)
- PDF에 있는 용어에 대해 선생님이 교과서 이외의 설명을 덧붙임
- 예: "이 개념은 쉽게 말하면 ~입니다", 예시나 비유를 추가

### 버킷 3 — 불확실 (사이드바)
- PDF 범위(지정한 페이지) 밖의 내용을 언급
- AI 확신도(confidence)가 낮은 항목
- 두 명 이상의 목소리가 겹치는 구간
- 전문 용어인데 STT가 잘못 인식했을 가능성이 있는 경우

---

## 🏗️ 핵심 설계 결정 (이미 확정된 것들)

### 결정 1 — 범위 제한
- 사용자가 "오늘 범위: N페이지 ~ M페이지"를 직접 지정
- AI는 이 범위의 PDF 텍스트 안에서만 강의 내용을 매핑
- 범위 밖 언급은 자동으로 버킷 3(불확실)으로 분류
- → **"선생님이 지금 어디를 설명하는지 추적" 문제를 해결**

### 결정 2 — 녹음 후 분석 (실시간 아님)
- 강의가 끝난 뒤 전체 녹음을 한 번에 분석
- 실시간 분석보다 정확도가 훨씬 높고, API 비용도 낮으며, 구현이 단순
- 나중에 실시간을 보조 기능으로 추가 가능

### 결정 3 — 웹 서비스
- 앱(iOS/Android) 아님, 데스크탑 전용 앱 아님
- 웹 브라우저에서 모두 동작
- → **설치 없이 바로 쓸 수 있어야 함**

## 🛠️ 기술 스택

### 프론트엔드
- **React 19** + TypeScript (strict mode)
- **PDF 뷰어**: PDF.js (Mozilla)
- **하이라이트/툴팁 오버레이**: PDF.js의 textLayer 위에 커스텀 오버레이
- **스타일링**: CSS (Tailwind 예정)
- **빌드**: Vite
- **라우팅**: React Router v7

### 백엔드 (예정)
- **프레임워크**: Python + FastAPI
- **파일 처리**: PyMuPDF (PDF 파싱 및 텍스트 좌표 추출)
- **음성 인식**: OpenAI Whisper API (또는 로컬 Whisper 모델)
- **AI 분석**: Claude API 또는 GPT-4o API
- **데이터베이스**: PostgreSQL (예정)
- **파일 저장**: AWS S3 또는 로컬 파일 시스템 (MVP)

## ✅ MVP 성공 기준

- **대상 수업**: 한국어 강의, 단일 주 강사, 30~60분 녹음
- **대상 PDF**: 텍스트 추출 가능한 교과서/강의자료 PDF, 사용자가 지정한 3~10페이지 범위
- **처리 시간 목표**: 45분 강의 기준 10분 이내 결과 제공
- **정확도 목표**: 자동 하이라이트의 70% 이상이 사용자가 납득 가능해야 함
- **UX 원칙**: 확실하지 않은 내용은 무리해서 PDF 위에 표시하지 않고 불확실 항목으로 보냄
- **첫 사용자 테스트 기준**: 사용자가 결과를 보고 5분 이내에 "그대로 쓸 것"과 "수정할 것"을 구분할 수 있어야 함

---

## 🚫 MVP에서 제외할 것

- 실시간 AI 분석
- 스캔 PDF OCR 자동 처리
- 여러 화자 분리(diarization)의 완전 자동화
- 모든 PDF 레이아웃에 대한 완벽한 좌표 매핑
- 학교 LMS 연동, 결제, 팀 협업, 모바일 앱
- 사용자가 업로드한 교재 원문을 모델 학습에 사용하는 기능

---

## 🎚️ 정확도 및 신뢰성 원칙

- 잘못된 위치에 자신 있게 표시하는 것보다 불확실 목록으로 보내는 것이 낫다.
- 모든 하이라이트/툴팁에는 PDF 페이지, 매칭된 원문, 전사 구간, confidence 근거가 있어야 한다.
- AI가 좌표를 직접 지어내게 하지 않는다. 좌표는 PDF 파서가 만든 text box id를 기준으로 연결한다.
- 버킷 1(강조)은 보수적으로 분류한다. 명시적 강조 표현, 반복, 전사 타임스탬프 근거가 없으면 버킷 2 또는 3으로 낮춘다.
- 버킷 2(추가 설명)는 PDF에 실제로 존재하는 용어와 연결될 때만 표시한다.
- 모델 출력은 항상 JSON schema로 검증하고, 실패 시 재시도 또는 fallback을 둔다.

---

## 🔐 개인정보, 저작권, 녹음 동의 원칙

- PDF와 오디오는 민감한 학습 자료일 수 있으므로 최소 저장, 자동 삭제, 사용자 삭제 기능을 기본값으로 설계한다.
- 실제 서비스에서는 수업 녹음에 대한 학교/교사/지역 법규상 동의 요건을 확인해야 한다.
- 미성년자 사용 가능성이 있으므로 계정, 보관 기간, 보호자 동의, 로그 마스킹 정책을 별도로 검토한다.
- 업로드된 교재 PDF를 외부에 공개하거나 공유하는 기능은 MVP에서 제외한다.

## 🌍 경쟁 서비스 분석

| 서비스 | 특징 | 한계 |
|--------|------|------|
| **Otter.ai** | 강의 전사 + 새 노트 문서 생성 | 원본 PDF 위 주석 아님 |
| **Notability** | iPad 앱, 필기와 오디오 싱크 | 웹 없음, 교과서 PDF 연동 없음 |
| **Coconote** | 강의 → 새 노트 생성 | PDF 뷰어 내 주석 없음 |
| **LectureNotesAI** | 강의 → 새 노트 생성 | 같은 한계 |

**결론**: "강의 오디오 → 원본 PDF 위에 직접 주석"을 하는 웹 서비스는 **현재 시장에 존재하지 않음**
