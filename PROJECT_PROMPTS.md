## 프로젝트 이름
Noto (LectureNote AI)

## 한 줄 설명
학생이 수업 전에 교과서 PDF와 오늘 나갈 페이지 범위를 올리고,
수업 중 녹음을 켜두면, 수업 후 AI가 원본 PDF 위에 직접 필기를 해주는 웹 서비스.

## 핵심 가치 제안 (Core Value Proposition)
기존 AI 노트 서비스들은 강의를 듣고 "새로운 노트 문서"를 만들어준다.
이 서비스는 다르다. 학생이 이미 갖고 있는 교과서 PDF 위에 직접 주석을 달아준다.
학생은 자기 교과서를 열었을 때, AI가 선생님 강의 내용을 바탕으로
필기를 대신 해놓은 것처럼 보이게 된다.

## 전체 사용자 플로우 (User Flow)

수업 전 (30초 세팅):
  1. 웹 브라우저에서 PDF 업로드
  2. "오늘 범위: 12페이지 ~ 18페이지" 입력
  3. 녹음 시작 버튼 클릭

수업 중:
  - 브라우저 탭을 열어두거나 닫아도 됨 (백그라운드 녹음)
  - 학생은 수업에만 집중

수업 후:
  1. 녹음 종료
  2. AI 분석 시작 (수 분 소요)
  3. 결과 화면: 원본 PDF 뷰어에서 결과 확인

## 결과 화면에서 보이는 것

PDF 뷰어 (메인 영역):
  - 원본 교과서 PDF가 그대로 표시됨
  - [노란 하이라이트]: 선생님이 강조한 용어나 문장 ("이거 시험에 나와요", 반복 언급)
  - [초록 하이라이트]: 선생님이 추가 설명을 한 용어 → 마우스를 올리면 툴팁으로 설명 표시
  - [회색 처리 / 표시 없음]: 선생님이 특별히 언급하지 않은 부분

사이드바 (오른쪽):
  - [불확실 항목 목록]: AI가 어디에 매핑해야 할지 확신하지 못한 내용들
  - 예: 선생님이 PDF 범위 밖의 내용을 언급한 경우
  - 예: AI가 용어를 정확히 인식하지 못한 경우
  - 학생이 직접 보고 수동으로 필기하거나 무시할 수 있음

## AI가 수행하는 분류 작업 (3가지 버킷)

버킷 1 — 강조 (노란 하이라이트):
  - 선생님이 "이거 중요해요", "시험에 나와요" 등 명시적으로 강조
  - 같은 용어를 2회 이상 반복
  - 목소리를 높이거나 속도를 늦추는 패턴 (STT 타임스탬프 기반 추정)

버킷 2 — 추가 설명 (툴팁):
  - PDF에 있는 용어에 대해 선생님이 교과서 이외의 설명을 덧붙임
  - 예: "이 개념은 쉽게 말하면 ~입니다", 예시나 비유를 추가

버킷 3 — 불확실 (사이드바):
  - PDF 범위(지정한 페이지) 밖의 내용을 언급
  - AI 확신도(confidence)가 낮은 항목
  - 두 명 이상의 목소리가 겹치는 구간
  - 전문 용어인데 STT가 잘못 인식했을 가능성이 있는 경우

## 핵심 설계 결정 (이미 확정된 것들)

결정 1 — 범위 제한:
  사용자가 "오늘 범위: N페이지 ~ M페이지"를 직접 지정.
  AI는 이 범위의 PDF 텍스트 안에서만 강의 내용을 매핑.
  범위 밖 언급은 자동으로 버킷 3(불확실)으로 분류.
  → 이 결정이 "선생님이 지금 어디를 설명하는지 추적" 문제를 해결함.

결정 2 — 녹음 후 분석 (실시간 아님):
  강의가 끝난 뒤 전체 녹음을 한 번에 분석.
  실시간 분석보다 정확도가 훨씬 높고, API 비용도 낮으며, 구현이 단순.
  나중에 실시간을 보조 기능으로 추가 가능.

결정 3 — 웹 서비스:
  앱(iOS/Android) 아님, 데스크탑 전용 앱 아님.
  웹 브라우저에서 모두 동작.
  → 설치 없이 바로 쓸 수 있어야 함.

## 기술 스택 (계획 중)

프론트엔드:
  - React 19 + TypeScript (strict mode)
  - PDF 뷰어: PDF.js (Mozilla)
  - 하이라이트/툴팁 오버레이: PDF.js의 텍스트 레이어 위에 커스텀 오버레이
  - 스타일링: CSS
  - 빌드: Vite
  - 라우팅: React Router v7

백엔드:
  - Python + FastAPI
  - 파일 처리: PyMuPDF (PDF 파싱 및 텍스트 좌표 추출)
  - 음성 인식: OpenAI Whisper API (또는 로컬 Whisper 모델)
  - AI 분석: Claude API 또는 GPT-4o API
  - 데이터베이스: PostgreSQL
  - 파일 저장: AWS S3 또는 로컬 파일 시스템 (MVP)

## 현재 상태
- 프론트엔드 부분 완성 (React 구조, UI/UX, 애니메이션)
- 백엔드 개발 예정
- 팀: 소규모 (1~2명)
- 목표: MVP 프로토타입 제작 → 실제 배포

## 경쟁 서비스 분석 (이미 조사 완료)
- Otter.ai: 강의 전사 + 새 노트 문서 생성. 원본 PDF 위 주석 아님.
- Notability: iPad 앱. 필기와 오디오 싱크. 웹 없음.
- Coconote, LectureNotesAI: 강의 → 새 노트 생성. PDF 뷰어 내 주석 없음.
- 한국 시장: 이 방식의 서비스 없음.
→ "강의 오디오 → 원본 PDF 위에 직접 주석"을 하는 웹 서비스는 존재하지 않음.

구현 요청 (우선순위 순서대로):

Module 1 — PDF 파싱 모듈 (가장 먼저)
  파일: backend/services/pdf_parser.py
  요구사항:
  - PyMuPDF(fitz)를 사용
  - 입력: PDF 파일 경로, 시작 페이지, 끝 페이지
  - 출력: 각 단어/문장의 텍스트와 PDF 좌표(x, y, width, height, page_number)
  - 한국어 텍스트 처리 (유니코드 정규화 포함)
  - 용어 추출: 페이지 범위 내 주요 명사/용어 리스트 별도 추출
  - 에러 처리: 스캔 PDF(이미지 PDF) 감지 및 사용자 안내 메시지 반환

Module 2 — AI 분석 모듈
  파일: backend/services/ai_analyzer.py
  요구사항:
  - TASK 02에서 설계한 프롬프트를 실제로 구현
  - 입력: PDF 텍스트 딕셔너리 + 강의 전사본(타임스탬프 포함)
  - 출력: 버킷별 분류 결과 JSON (TASK 02에서 설계한 포맷)
  - API 호출 실패 시 재시도 로직 포함
  - 긴 전사본을 청크로 나누어 처리하는 로직 포함 (컨텍스트 윈도우 초과 방지)
  - confidence score 기반으로 버킷 3(불확실) 자동 분류

Module 3 — FastAPI 엔드포인트
  파일: backend/main.py
  요구사항:
  - POST /upload-pdf: PDF 파일 + 페이지 범위 받기
  - POST /upload-audio: 녹음 파일 받기 (mp3, wav, m4a 지원)
  - POST /analyze: 분석 시작 (비동기 처리)
  - GET /result/{session_id}: 분석 결과 폴링
  - 세션 관리: 각 수업마다 고유 session_id 발급
  - CORS 설정 (프론트엔드 로컬 개발 환경 허용)
